# 踩坑说明

这份文档我主要是想把这个项目是怎么一步一步做出来的记下来。

我想记的不是功能清单，而是这些东西：

- 我一开始是怎么想的
- 我中间为什么改思路
- 哪些路我试过了，但最后没有继续
- 我最后为什么收敛成现在这套结构

## 一开始我想做的其实很直接

我的目标最开始很简单：

1. 网页能输入文本
2. 服务端把文本做成 TTS
3. ESP32 把声音播出来

真正做起来以后，我很快发现问题不在“能不能播”，而在“能不能稳定播”。

## 我先做的是最基础的播放链路

我没有一上来就先做 TTS，而是先把最基础的播放链路跑通：

```text
网页
-> PHP
-> MQTT
-> ESP32
-> 播放固定 URL 音频
```

我先这么做，主要是为了先确认几件事：

- MQTT 命令能不能发到设备
- ESP32 能不能正常拉 `http://` 音频
- I2S、功放、喇叭是不是都正常

这个阶段踩到的坑也很典型：

- MQTT 看起来连上了，但其实不是同一个 broker
- `127.0.0.1` 这种地址对 ESP32 没意义
- HTTPS 地址不能直接播
- M4A 不能直接当 MP3 解码

也就是从这一步开始，我慢慢意识到这个项目真正麻烦的地方，不是页面，也不是接口，而是整条链路每一层都要对上。

## 这版我直接用的是火山引擎

当前这版我直接用火山引擎来做 TTS。

我当时的思路很明确：

- Python 这一层专门负责对接 TTS
- 网页、PHP、ESP32 的控制链尽量保持清楚
- 尽量把“文本转语音”和“设备播放”拆开

最后保留下来的主链路就是：

```text
网页
-> PHP
-> Python Volcengine TTS bridge
-> MQTT
-> ESP32
-> HTTP 实时拉流播放
```

## 我为什么没有走 MQTT 传音频

这个方向我认真想过。

理论上是可以这样做的：

```text
Python 拿到音频块
-> 通过 MQTT 发给 ESP32
-> ESP32 解码播放
```

但我最后没有这么做，因为我觉得当前阶段不划算。

原因很直接：

1. MQTT 直接传音频，会让 ESP32 这一层复杂很多
2. 我得自己处理缓冲、连续字节流、切流、清缓冲这些事
3. 这已经不是“补功能”，而是接近“重做一套播放器”
4. 对我这版来说，HTTP 实时流已经足够接近目标

所以我最后保留的是：

**HTTP 实时流 + MQTT 控制通道分离**

现在回头看，这个决定我觉得是对的。

## 我为什么没继续坚持整段长文本一次性播完

最开始我当然是想一整段直接播完的。

比如：

- 一整段文本
- 一次 TTS
- ESP32 一次性完整播出来

但实际一做就发现问题很明显：

- 文本越长，播放时间越长
- 播放时间越长，ESP32 越容易撞上 WiFi 抖动
- 一旦 WiFi 抖动，HTTP 音频流就断
- 最后就会出现 `stream_interrupted`

所以我后面越来越明确一件事：

**长文本的问题不是“不能生成”，而是“单条流持续太久时更容易被网络抖动打断”。**

也就是说，这已经不是单纯的 TTS 问题，而是 ESP32 网络稳定性问题。

## 我为什么没有把插播当成这版重点

插播我一开始也很想做。

理想状态当然是：

- 第一条正在播
- 第二条来了马上插进去
- 立刻切到下一条

但真调的时候我发现：

- ESP32 在播放 HTTP 流时，MQTT 并不总是稳定
- 第二条命令经常在播放期间根本到不了设备

所以问题并不是“我没写插播逻辑”，而是：

**当前这版在 ESP32 这层，同时保持 MQTT 控制和 HTTP 拉流播放，稳定性还不够支撑强插播。**

所以我后面就不再把“播放中抢占式插播”当成当前版本的主要目标了。

## PHP 分段顺序播报是怎么来的

既然整段长流容易断，我就开始往“分段顺序播报”这个方向收。

最早的版本是：

```text
播完一段
-> PHP 再去生成下一段 TTS
-> 再通过 MQTT 下发
```

这样确实稳了不少，但新问题马上就出来了：

- 段间空档比较明显

后面我继续把它改成：

```text
PHP 先把所有分段都预生成好
-> 先发第一段
-> ESP32 播完一段
-> PHP 立刻发下一段
```

这样就把“段间现生成 TTS”的时间省掉了，体感上会更顺一点。

## 我为什么把断句放在 PHP

断句我最后放在了 PHP，而不是前端，也不是 Python。

我当时的考虑是：

1. PHP 本身就在网页请求和设备控制之间
2. 前端没必要理解设备播放策略
3. Python 主要负责对接 TTS，职责更单一一点更好
4. PHP 最适合做“断句、分段、排队、顺序下发”

所以最后职责就比较清楚了：

- 前端：发请求
- PHP：断句、分段、预生成、顺序下发
- Python：生成流式 TTS
- ESP32：只负责播放

## 我最后收敛出来的方案

当前这版最后收敛到的是：

```text
用户文本
-> PHP 自动断句分段
-> PHP 预生成每一段的 TTS stream_url
-> MQTT 下发 tts + url
-> ESP32 打开 HTTP stream
-> AudioFileSourceBuffer 本地缓冲
-> I2S 播放
-> MQTT 继续负责状态和心跳
```

这里面几个关键点我现在是比较确定的：

- TTS 格式优先 `mp3`
- `sample_rate` 现在主用 `24000`
- ESP32 用 `AudioFileSourceBuffer`
- 当前缓冲用 `32KB`
- MQTT 不传音频，只传命令
- 长文本不再整段硬播，而是分段顺序播放
- PHP 提前把每一段先生成好，减少段间等待

## 我现在觉得真正解决了什么

做到现在，我觉得这版真正解决的是：

- 页面可以输入任意文本
- 火山引擎流式 TTS 能正常工作
- ESP32 能稳定拉流播放短文本
- 长文本不再依赖一条长流硬撑到底
- 长文本已经能通过分段顺序播报的方式稳定很多

## 现在还留着的边界

### 1. 还不是完全无缝

虽然已经做了预生成，但多段之间还是会有一点停顿。

这不是我没注意到，而是当前结构本身就是一段一个流、一段一个命令，所以不可能完全无缝。

### 2. WiFi 环境还是很关键

日志已经反复证明了：

- 如果 ESP32 到路由器这一层本身有抖动
- 单段播放本身也可能受影响

所以到现在这个阶段，我已经不把这看成纯代码问题了，它和现场网络环境也直接相关。

### 3. 这版不主打强插播

这版更适合：

- 单条播报
- 长文本顺序播报
- 多段连续播报

而不是复杂的抢占式插播。

## 这次我最大的感受

这个项目做到后面，我最大的感受不是“功能终于堆出来了”，而是：

**任何一层没确认清楚，最后都会看起来像别的层出了问题。**

这次我反复碰到的其实就是这些：

- 页面到底有没有把请求发出去
- PHP 到底有没有真的发 MQTT
- Python 到底有没有真的生成 TTS
- ESP32 到底有没有收到命令
- ESP32 到底有没有打开流
- WiFi 到底有没有中途抖动
- I2S 和功放到底有没有真的在播

也正因为我把这些一层一层拆开，最后才慢慢收敛到现在这套方案。

## 对我来说，这个版本的意义

这个版本对我来说，最大的意义不是“它完美了”，而是：

**它已经是一条真正跑通，而且经过一轮轮踩坑后收敛出来的完整链路。**

也就是说，到现在这个项目已经不是一个想法了，而是一个已经能工作的版本。
